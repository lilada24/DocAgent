# Part 3: Let AI Safely Edit Your Code

The design I admire the most in Claude Code isn't some sophisticated architectural pattern, but a seemingly simple constraint in `FileEditTool`: **old_string must appear uniquely in the file.**

This single rule solves one of the most troublesome problems in the field of AI code editing.

---

## Three Dead Ends in Code Editing

The industry tried many approaches to let LLM edit code, but each of them has its pitfalls.

**Approach 1: Line Number Patching** ("Change line 42 to xxx").

This seems the most direct, but is actually the least reliable. LLM's ability to remember line numbers is very poor—especially after context compression, the line numbers it remembers may no longer match. Even worse is the concurrent scenario: if LLM inserts a line at line 30, all subsequent line numbers change. If the next editing operation still uses the old line numbers, it will modify the wrong place.

This is not a low-probability event. It will almost certainly occur during multiple rounds of editing.

**Approach 2: Rewrite the Entire File** ("Output a complete new file").

The most brute-force and safest option. The problem is cost. For a 500-line file, if you only want to modify 2 lines, LLM has to regenerate all 500 lines. A GPT-4o output token costs approximately $15/MB, so 500 lines would cost around 2000 tokens, resulting in $0.03 per edit. If a task requires modifying 20 files, each requiring multiple rounds of modification… the cost explodes.

Furthermore, LLM frequently makes mistakes when "copying" long text—dropping lines, changing formatting, turning tabs into spaces. You might only want to change a function signature, but the indentation of the entire file becomes messed up.

**Approach 3: Unified Diff Format** ("Output a diff patch").

Diff is the standard format for human code review and is theoretically the most efficient. However, the accuracy of diff formats generated by LLM is very low. Line number offsets are frequently miscalculated, context line numbers are incorrect, and sometimes even the `+` and `-` prefixes are messed up. You need a very complex fault-tolerant parser to handle these formatting errors, and even if the parser succeeds, it can't detect the errors when applying the parser to the correct location. Aider leaned heavily on this approach, using extensive prompt engineering to get the LLM to output a standard diff format, but it still couldn’t completely eliminate formatting errors.

## Claude Code's Solution: Search and Replace

Claude Code chose Approach 4.

LLM doesn't need to know the line numbers, doesn't need to output the entire file, and doesn't need to generate a diff format. It only needs to do two things:

1. Provide a **precise**, **unique** text (old_string) that exists only in the file.
2. Provide the text to be replaced (new_string)

```json
{
  "file_path": "src/auth.py",
  "old_string": "def verify_token(token):\n    return jwt.decode(token, SECRET)",
  "new_string": "def verify_token(token):\n    try:\n        return jwt.decode(token, SECRET)\n    except jwt.ExpiredSignatureError:\n        return None"
}
```

Key constraint: **old_string must appear exactly once in the file.**

- 0 times → LLM misremembered the file content. Returns an error with a preview of the beginning of the file, allowing it to `read_file` and try again.
- 2 times or more → Insufficient context to determine which part to modify. Returns an error, requesting the LLM to include more surrounding lines to disambiguate.
- Exactly 1 time → Deterministic replacement. It's impossible to change the wrong position.

This constraint is clever as it transforms the inherently ambiguous operation of "editing a file" into a **deterministic** operation. No matter how poor the LLM's memory is, as long as it provides a unique piece of text, the edit will be accurate.

CoreCoder fully implements this pattern (`tools/edit.py`, line 70), and adds unified diff output—after each edit, you can see a comparison of changes in the format `--- a/file +++ b/file`.

## Tool Interface: buildTool() Factory

Those who have worked on Agent frameworks may have written a `BaseTool` base class and then extended it. Claude Code doesn't inherit from it at all.

All tools are plain objects generated by the `buildTool()` factory function, with their types defined in `src/Tool.ts` (line 792):

```typescript
// src/Tool.ts (simplified)
type ToolDef<Input, Output> = {
  name: string
  description: string
  inputSchema: ZodSchema<Input>        //  Zod for validation + automatic JSON schema generation
  call(input: Input, ctx: ToolUseContext): AsyncGenerator<ToolProgress | ToolResult>

  // Security
  isReadOnly(): boolean
  getPermissions(input: Input): ToolPermission[]
  validateInput?(input: Input, ctx: ToolUseContext): ValidationResult
  checkPermissions(input: Input, ctx: ToolUseContext): PermissionResult

  // UI
  renderToolUse?(input: Input): ReactNode
  renderToolResult?(result: Output): ReactNode

  // Context economy
  getToolUseSummary?(input: Input, result: Output): string
  maxResultSizeChars: number
}
```

Several fields worth noting:

**`getToolUseSummary()`:** this method determines how a tool call is summarized when the context is compressed. For example, the summary for `BashTool` might be "ran: npm test (exit code 0)", compressing 500 lines of output into a single sentence. Different tools know which parts of their output are important, so their summarization strategies differ.

**`maxResultSizeChars`:** Output exceeding this threshold is written to disk, leaving only a file path reference and summary in the context. Setting it to `Infinity` means never writing to disk. `GrepTool` has a lower threshold than `ReadFileTool` because search results are typically longer.

**`renderToolUse()` returns a ReactNode:** Yes, the tool's terminal rendering component is a React component. The tool calls in the terminal feature a beautiful progress bar, code highlighting, and diff preview, all rendered using React + Ink.

## Two-Phase Gating: Why Permission Pop-ups Aren't So Annoying

Claude Code's permission pop-ups are probably the least annoying of all AI programming tools. The reason is two-phase gating:

```
用户说："删掉 temp/ 目录"
  │
  ▼
validateInput()
  ├─ Path exists? → No → Directly tell LLM "path does not exist", no pop-up
  ├─ Parameter valid? → No → Directly tell LLM "parameter error", no pop-up
  └─ Validated 
        │
        ▼
    checkPermissions()
        ├─ Current permission level allows? → Yes → Execute directly
        └─ Confirmation required? → Pop-up window asks user
```

Most rejected tool calls are due to invalid input (file does not exist, path format is incorrect, parameter type is incorrect), not insufficient permissions. The first stage handles these without pop-ups, only disturbing the user when a permission decision is actually needed.

Claude Code has five permission modes: fully automatic, automatic for recommended operations, confirmation required for write operations, confirmation required for all operations, and read-only mode. Enterprise administrators can enforce these settings through configuration files.

## BashTool: A 1143-lines security fortress

BashTool is a single file of 1143 lines. It does much more than `subprocess.run(command)`:

**Command classifier.** Each command is parsed into three categories: search / read / write. `grep` is search, `cat` is read, and `rm` is write. The classification results are used to match permission rules—the user may have allowed all search and read operations, but write operations require confirmation for each one. Compound commands (`ls && git push`) are broken down and evaluated segment by segment.

**Sandboxing.** On macOS, `sandbox-exec` is used; on Linux, `seccomp` is used. Sandboxing restricts network access, file system access, etc.

**Large Output Management.** Command output is truncated when it exceeds a threshold, retaining only the header and footer. The header typically includes command echo and initial results, while the footer usually contains summary information and error messages. Repeated output in the middle is not very useful for LLM's decision-making.

**Interactive Command Interception.** Commands requiring user interaction, such as `vim`, `less`, `ssh`, and `python` (without parameters), are rejected outright. LLM does not operate the interactive terminal.

**Timeout and Background Transition.** Commands exceeding 15 seconds are automatically transferred to the background, allowing LLM to continue other tasks.

**sed Detection.** When the `sed -i` command is detected, the UI switches from a "Bash" style to a file editing style. Because `sed -i` essentially edits a file, it should display a diff like `edit_file`.

CoreCoder's BashTool is a streamlined 80-line version: output truncation (preserving the beginning and end), timeout control, and regular expression checks for 9 dangerous commands (`rm -rf /`, fork bomb, `curl | bash`, etc.). It covers the most core security requirements.

## Dynamic Tool Assembly

Claude Code does not have a global tool registry. The tool pool for each session is dynamically assembled:

```typescript
// Assemble the tool list at the start of the session
const tools = [
  ...builtinTools,      // 30+ built-in tools
  ...mcpTools,          // Tools discovered from the MCP server
  ...agentDefinedTools, // Custom tools defined by the Agent
  ...skillTools,        // Tools exposed by the Skill
]
```

Some tools are "lazy-loaded"—tools marked `shouldDefer: true` are not in the tool list by default; they are only discovered and loaded when the LLM triggers `ToolSearch`. This avoids the LLM's difficulty in selecting tools due to an excessively long tool list (too many tools will cause the LLM to hesitate, affecting response speed and accuracy).

CoreCoder uses the simplest static list. However, the interface design leaves room for expansion—`Agent.__init__` accepts a `tools` parameter; whatever is passed is used.
